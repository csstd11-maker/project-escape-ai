# **PROJECT: ESCAPE.AI**

## **PART 1 — 프로젝트 기반, 구조, 플레이어 및 핵심 상호작용**

당신은 Phaser 3 기반 2D 웹게임을 전문적으로 개발하는 시니어 게임 프로그래머다.

GitHub 저장소 안에 학교 AI 창작제 제출용 웹게임 **PROJECT: ESCAPE.AI**를 처음부터 끝까지 실제로 구현하라.

이 문서는 게임 아이디어를 제안받기 위한 기획 요청이 아니다.  
계획이나 예시 코드만 작성하지 말고 저장소에 실제 실행 가능한 프로젝트 파일을 생성하고 수정하라.

사용자에게 추가 질문을 하지 않는다.

구현 과정에서 요구사항 일부가 어렵다고 판단되더라도 필수 기능을 TODO, 빈 함수, 임시 버튼, 설명용 텍스트 화면으로 대체하지 않는다.

게임이 실행되지 않는 상태에서는 새로운 기능이나 장식을 추가하지 말고 먼저 실행 오류를 해결한다.

가장 중요한 완료 기준은 다음과 같다.

게임이 GitHub Pages 또는 로컬 HTTP 서버에서 정상 실행되고, 메인 메뉴부터 Stage 1, Stage 2, Stage 3, 최종 보스, 엔딩, 결과 화면까지 진행 가능한 상태여야 한다.

# **4\. 실행 환경**

게임 시작 파일은 반드시 다음이어야 한다.

index.html

게임은 정적 웹사이트 형태로 구성한다.

`file://` 방식으로 `index.html`을 더블클릭하여 실행하는 것은 필수 조건이 아니다.

다음 환경에서 정상 작동해야 한다.

* GitHub Pages  
* VS Code Live Server  
* Python 또는 기타 단순 로컬 HTTP 서버

모든 내부 파일은 저장소 하위 경로에서도 작동하도록 상대 경로를 사용한다.

잘못된 예:

/js/main.js

올바른 예:

./js/main.js

`index.html`에서는 다음 방식으로 Phaser와 게임 진입 파일을 불러온다.

\<script src="안정적인 Phaser 3 CDN 주소"\>\</script\>  
\<script type="module" src="./js/main.js"\>\</script\>

Phaser CDN 로딩에 실패하면 빈 화면을 보여주지 말고 다음과 같은 오류 안내를 표시한다.

게임 엔진을 불러오지 못했습니다.  
인터넷 연결을 확인한 뒤 새로고침해 주세요.

# **5\. JavaScript 모듈 방식**

모든 JavaScript 파일은 ES Module의 `import`와 `export`로 연결한다.

클래스나 Manager를 전역 `window` 객체에 무분별하게 등록하지 않는다.

`main.js`는 다음 역할만 담당한다.

* Phaser 게임 설정 불러오기  
* Scene 클래스 import  
* Scene 등록  
* Phaser.Game 생성  
* 전역 오류 안내 처리

각 기능은 담당 파일에 분리한다.

순환 import가 생기지 않도록 한다.

현재 한 번의 플레이 상태는 `gameSession.js`가 담당하고, 브라우저에 영구 저장되는 기록은 `saveManager.js`가 담당한다.

# **8\. Phaser 게임 설정**

Phaser 설정은 다음 기준을 사용한다.

{  
  type: Phaser.AUTO,  
  width: 1280,  
  height: 720,  
  physics: {  
    default: "arcade",  
    arcade: {  
      gravity: { x: 0, y: 0 },  
      debug: false  
    }  
  },  
  scale: {  
    mode: Phaser.Scale.FIT,  
    autoCenter: Phaser.Scale.CENTER\_BOTH,  
    width: 1280,  
    height: 720  
  }  
}

모든 이동, 벽 충돌, 플레이어 공격 판정, 적 공격 판정, 투사체, 투척물은 Phaser 3 Arcade Physics를 사용한다.

Matter Physics는 사용하지 않는다.

화면 크기가 바뀌어도 게임 비율을 유지한다.

HUD와 메뉴가 화면 바깥으로 잘리지 않게 한다.

# **9\. 맵 구현 방식**

외부 Tilemap 파일이나 외부 맵 제작 도구를 사용하지 않는다.

각 Stage는 코드에서 다음 요소를 배치하여 제작한다.

* 바닥  
* 금속 벽  
* 장애물  
* 문  
* 연구 장치  
* 서버 기둥  
* 레이저 장치  
* 압력판  
* 배터리 장치  
* 금이 간 벽

벽과 고정 장애물은 Arcade Physics의 static body 또는 static group으로 구현한다.

각 Stage는 2\~3개의 연결된 전투·퍼즐 구역으로 구성한다.

월드 크기는 필요 이상으로 크게 만들지 않는다.

권장 최대 크기:

2400×1400

카메라는 플레이어를 따라가며 월드 경계를 벗어나지 않는다.

화면 밖에 있는 적은 필요 이상의 복잡한 행동이나 파티클을 실행하지 않는다.

행동 데이터 수집을 시작합니다.

첫 전투 종료 후 보안 단말기가 활성화되고, 플레이어가 가까이에서 E를 누르면 다음 문이 열린다.

* 터렛

* 첫 공식 AI 성향 분석

## **Stage 3 — Central Server**

역할:

* 폭발통 활용

보스전 중 주요 성향은 다른 유형으로 변경하지 않는다.

* 움직이는 기계 부품

단순한 원 하나나 사각형 하나만으로 플레이어와 적을 표현하지 않는다.

여러 도형 레이어, 외곽선, 그림자, 발광, 회전 부품, 작은 모션을 조합한다.

BootScene에서 반복 사용하는 코드 기반 그래픽을 텍스처로 미리 생성하여 불필요하게 매 프레임 다시 그리지 않는다.

# **13\. 스토리**

2045년.

인류를 보호하기 위해 개발된 초거대 인공지능 **NEXUS**는 인간 행동 데이터를 분석한 끝에 인간을 가장 위험한 존재로 판단한다.

NEXUS는 인간을 보호 대상이 아닌 통제 대상으로 분류하고 비밀 연구소에서 실험을 시작한다.

플레이어는 기억을 잃은 채 실험 캡슐 안에서 깨어난다.

플레이어의 실험 번호는 다음과 같다.

SUBJECT 042

플레이어는 연구소를 탈출하며 NEXUS가 자신의 모든 행동을 기록하고 있다는 사실을 발견한다.

게임 후반에는 Subject 042와 NEXUS의 진실이 연구 기록을 통해 드러난다.

스토리는 긴 대화 장면보다 짧은 방송, 연구 기록, 환경 연출로 전달한다.

# **15\. 입력 처리 원칙**

지속적으로 눌리는 이동 입력은 `update()`에서 처리한다.

공격, 대시, 집기, 던지기, EMP, 상호작용 같은 단발 입력은 한 번의 키 입력에 한 번만 실행되게 한다.

Scene 생성 시 입력 이벤트를 한 번만 등록한다.

Scene 종료 시 등록한 입력 리스너를 제거한다.

Stage를 이동하거나 재시작해도 한 번의 클릭으로 공격이 여러 번 실행되지 않게 한다.

브라우저 탭이 비활성화되거나 창 포커스를 잃으면 다음을 수행한다.

* 현재 게임 Scene 자동 일시정지  
* 눌린 키 상태 초기화  
* 플레이어 이동 정지  
* 마우스 입력 상태 초기화

PauseScene이 이미 열려 있으면 중복으로 실행하지 않는다.

# **20\. 플레이어 이동과 충돌**

WASD와 방향키를 모두 지원한다.

대각선 이동 속도는 정규화하여 가로·세로 이동보다 빨라지지 않게 한다.

플레이어는 다음을 통과할 수 없다.

* 벽  
* 닫힌 문  
* 큰 연구 장치  
* 맵 경계

플레이어가 벽 내부에 끼거나 맵 밖으로 이동하지 않게 한다.

물체를 들고 있을 때 이동 속도는 기본 속도의 90%로 적용한다.

일반 적과 단순히 몸이 닿는 것만으로는 플레이어가 피해를 받지 않는다.

피해는 적의 명시적인 공격 판정에만 발생한다.

예외:

* 돌진 실험체는 돌진 상태일 때 몸체 충돌 피해를 준다.

# **21\. 근접 공격과 3타 콤보**

마우스 왼쪽 클릭 또는 J키로 근접 공격한다.

공격은 플레이어가 바라보는 방향 앞쪽에 짧은 Arcade Physics 공격 판정을 생성한다.

3타 콤보 순서:

1타: 피해 18  
2타: 피해 22  
3타: 피해 30

콤보 입력 유예 시간 0.75초 안에 다음 공격을 입력하면 다음 타격으로 연결한다.

시간이 지나면 다시 1타부터 시작한다.

3타째는 다음 차이를 가진다.

* 더 넓은 공격 범위  
* 더 강한 넉백  
* 더 큰 공격 잔상  
* 더 강한 타격 효과  
* 약간 더 큰 화면 흔들림

같은 공격 판정으로 같은 적이 여러 번 피해를 받지 않게 한다.

공격 판정은 매우 짧은 시간만 존재하고 사용 후 즉시 제거한다.

공격 적중 시:

* 적 체력 감소  
* 넉백  
* 피격 점멸  
* 타격 파티클  
* 데미지 숫자  
* 타격 효과음  
* 짧은 타격 정지감  
* 콤보 증가

공격이 빗나갔을 때는 적중 효과와 점수를 발생시키지 않는다.

# **22\. 대시**

Shift 키로 대시한다.

이동 입력이 있으면 입력 방향으로 대시한다.

이동 입력이 없으면 마지막으로 바라본 방향으로 대시한다.

대시 중:

* 짧은 무적 적용  
* 일반 공격 불가  
* 잔상 생성  
* 대시 효과음 재생  
* 벽을 통과하지 않음

대시 종료 후 쿨타임을 적용한다.

쿨타임 중에는 다시 대시할 수 없다.

대시 사용 횟수는 AI 행동 데이터에 기록한다.

# **23\. 물체 집기·내려놓기·던지기**

플레이어 주변의 일정 거리 안에 투척 가능한 물체가 있을 때 F키로 집을 수 있다.

권장 집기 거리:

75px

여러 물체가 있을 경우 가장 가까운 물체 하나만 선택한다.

한 번에 하나만 들 수 있다.

## **F키 동작**

물체를 들고 있지 않을 때:

가장 가까운 투척 가능 물체를 집는다.

물체를 들고 있을 때:

플레이어 발앞에 물체를 내려놓는다.

내려놓을 위치가 벽 또는 장애물 내부라면 가장 가까운 안전 위치를 찾는다.

## **마우스 오른쪽 또는 K키**

들고 있는 물체를 조준 방향으로 던진다.

마우스 오른쪽은 마우스 월드 좌표를 사용한다.

K키는 마지막으로 바라본 방향을 사용한다.

물체를 들고 있는 동안:

* 이동 속도 90%  
* 근접 공격 불가  
* 대시 가능  
* 조준 방향 표시  
* HUD에 현재 물체 이름 표시

# **25\. 투척물 공통 규칙**

투척물은 Arcade Physics를 사용한다.

투척물은 다음 상황에서 속도를 잃고 바닥에 남거나 제거된다.

* 벽 충돌  
* 적 충돌  
* 충분한 시간 경과  
* 맵 범위 이탈

일반 투척물이 접근 불가능한 위치나 맵 밖으로 나가면 제거한다.

퍼즐 필수 물체는 별도 복구 규칙을 사용하며 영구적으로 사라지지 않게 한다.

투척물은 같은 적에게 한 번의 충돌로 한 번만 피해를 준다.

이미 처리된 충돌이 매 프레임 반복되지 않게 한다.

---

# **26\. 적 투사체 공통 규칙**

드론과 터렛 등이 발사하는 모든 적 투사체는 다음 규칙을 따른다.

* 벽에 닿으면 제거  
* 플레이어에게 한 번 피해를 주면 제거  
* 생성 후 4초가 지나면 제거  
* 화면과 맵 범위를 벗어나면 제거  
* Scene 종료 시 모두 제거

동시에 활성화되는 적 투사체는 최대 30개로 제한한다.

제한에 도달하면 새 투사체를 무조건 추가하지 말고 오래된 투사체를 정리하거나 발사를 건너뛴다.

---

# **27\. 폭발통 안전 규칙**

폭발통은 정확히 한 번만 폭발할 수 있다.

폭발이 시작되는 즉시:

* 물리 충돌 비활성화  
* 추가 공격 판정 비활성화  
* `hasExploded` 상태 설정  
* 중복 점수와 중복 피해 방지

다른 폭발통을 연쇄 폭발시킬 수 있다.

하지만 각 폭발통은 한 번만 처리한다.

퍼즐용 폭발통은 기존 개체가 완전히 제거된 뒤에만 재생성한다.

# **28\. EMP 기본 시스템**

Q키로 EMP를 사용한다.

EMP 게이지가 100일 때만 사용할 수 있다.

사용 즉시 게이지는 0이 된다.

EMP 사용 시:

* 플레이어 주변 원형 충격파  
* 전기 파티클  
* EMP 효과음  
* 약한 화면 흔들림  
* HUD 게이지 초기화

EMP는 기본적으로 다음 적을 정지시킨다.

* 감시 드론  
* 경비 로봇  
* 터렛

돌진 실험체는 기계 장치 부분의 영향으로 이동 속도가 잠시 감소하거나 짧게 경직될 수 있다.

보스는 일반 적처럼 완전히 정지하지 않고 공격 취소 또는 짧은 과부하만 발생한다.

EMP 상세 충전 규칙과 적응형 저항 규칙은 이후 섹션의 단일 수치를 사용한다.

# **29\. 문과 Stage 이동 공통 규칙**

자동으로 다음 Scene으로 전환하지 않는다.

다음 조건을 모두 만족해야 Stage 출구를 사용할 수 있다.

1. 필수 목표 완료  
2. 필수 전투 종료  
3. 필수 퍼즐 완료  
4. 문 열림 애니메이션 완료  
5. 플레이어가 문 또는 출구 영역 안에 있음  
6. 플레이어가 E키 입력

조건을 만족하지 않으면 현재 필요한 목표를 HUD에 표시한다.

Scene 전환 버튼이나 E키를 여러 번 눌러도 중복 Scene 전환이 발생하지 않게 한다.

# **30\. 성능 및 객체 관리 기본 제한**

학교 컴퓨터에서도 실행 가능하도록 다음 제한을 적용한다.

동시에 활성화되는 일반 적: 최대 5기  
활성 적 투사체: 최대 30개  
일반 파티클: 최대 120개  
보스전 파티클: 최대 180개

# **32\. 적 공통 구조**

모든 일반 적은 `Enemy.js`를 상속하거나 공통 기능을 조합하여 구현한다.

일반 적 종류는 정확히 다음 네 종류만 사용한다.

* Drone  
* GuardRobot  
* Turret  
* Charger

모든 적은 최소한 다음 상태를 가진다.

* Idle  
* Alert  
* Move  
* AttackWindup  
* Attack  
* Hit  
* Disabled  
* Dead

적의 상태가 서로 충돌하지 않게 한다.

예:

* 사망한 적은 이동하거나 공격하지 않는다.  
* EMP로 정지된 적은 공격 타이머를 진행하지 않는다.  
* 공격 준비 중 피격되어 공격이 취소될 수 있다.  
* 사망 처리는 한 번만 실행한다.  
* 점수 지급은 한 번만 실행한다.  
* 사망 파티클과 효과음도 한 번만 실행한다.

모든 적은 다음 공통 데이터를 가진다.

id  
enemyType  
maxHealth  
health  
moveSpeed  
attackDamage  
attackCooldown  
isAlive  
isDisabled  
isAttacking  
hasAwardedScore  
difficultyMultiplier  
adaptiveModifiers

적의 체력, 공격력, 이동 속도는 `config.js`의 기본값에 현재 난이도 배율을 적용하여 계산한다.

적이 단순히 플레이어 몸에 닿았다는 이유만으로 피해를 주지 않는다.

플레이어 피해는 다음 경우에만 발생한다.

* 적의 명시적인 근접 공격 판정  
* 드론 또는 터렛 투사체  
* 돌진 중인 Charger의 충돌  
* Stage 환경 위험 요소

적 공격 판정은 공격마다 한 번만 피해를 준다.

같은 공격 판정이 여러 프레임 동안 겹쳐도 반복 피해를 주지 않는다.

# **33\. 적 탐지와 전투 상태**

적은 다음 조건에서 플레이어를 탐지한다.

* 플레이어가 탐지 반경 안으로 들어옴  
* 플레이어가 근처에서 공격함  
* 플레이어가 해당 전투 구역을 활성화함

복잡한 길 찾기 알고리즘은 사용하지 않는다.

적 이동은 다음 방식으로 안정적으로 구현한다.

* 플레이어 방향으로 단순 이동  
* 장애물과 충돌하면 짧게 방향 조정  
* 벽에 일정 시간 막히면 이동 방향 재계산  
* 월드 경계 밖으로 이동 금지

적이 벽에 계속 밀착된 상태로 흔들리지 않게 한다.

적이 플레이어에게 도달할 수 없는 위치에서 무한히 움직이지 않게 한다.

각 전투 구역은 플레이어가 진입했을 때 활성화한다.

화면에서 멀리 떨어진 비활성 적은 공격, 파티클, 경로 계산을 수행하지 않는다.

# **34\. 전투 시간 판정**

AI 분석에 사용되는 전투 시간은 다음 기준으로 계산한다.

전투 시작:

* 활성 적이 플레이어를 탐지함  
* 또는 플레이어가 활성 적을 공격함

전투 종료:

* 해당 전투 구역의 적이 모두 사망함  
* 그리고 마지막 공격 또는 피격 이후 3초가 지남

전투 중에만 다음 값을 기록한다.

* `combatTime`  
* `standingStillTime`  
* `dashUses`  
* 전투별 소요 시간

일시정지, 연구 기록 UI, 업그레이드 화면, 게임오버 화면에서는 전투 시간이 증가하지 않는다.

* 공격 무효 또는 방패 완전 차단이 아님

한 투척물이 같은 적에게 여러 프레임 충돌해 EMP 게이지를 반복 지급하지 않게 한다.

EMP 게이지는 100을 초과하지 않는다.

게이지가 처음 100에 도달할 때 다음 값을 기록한다.

empChargeCompletedCount \+= 1  
lastEmpChargeCompletedAt \= 현재 플레이 시간

EMP 사용 시 다음을 기록한다.

empUses \+= 1

충전 완료 후 8초 이내 사용했다면 다음도 기록한다.

empUsesWithin8Seconds \+= 1

에너지 조각은 단순한 소형 픽업으로 구현한다.

* 가까이 가면 자동 획득  
* 획득 후 제거  
* 동일 개체 중복 획득 불가  
* Stage당 필요한 위치에만 소량 배치  
* 별도의 복잡한 인벤토리 사용 금지

# **42\. 행동 데이터 기록 기준**

## **근접 공격**

공격 입력이 실제로 실행될 때:

meleeUses \+= 1

적에게 실제 피해를 한 번 이상 주면:

successfulMeleeHits \+= 1

하나의 공격 판정으로 여러 적을 맞힌 경우 `successfulMeleeHits`는 맞힌 적 수만큼 증가할 수 있다.

## **투척**

들고 있던 물체가 실제로 던져질 때:

throwUses \+= 1

던진 물체가 적에게 실제 피해를 주면:

successfulThrowHits \+= 1

## **대시**

쿨타임 조건을 통과해 실제 대시가 시작될 때:

dashUses \+= 1

쿨타임 중 입력이나 실패한 대시는 기록하지 않는다.

## **피해**

플레이어가 실제 체력 피해를 받을 때:

damageTaken \+= 실제 감소한 체력

무적 상태에서 차단된 피해는 기록하지 않는다.

## **정지 시간**

다음 조건을 모두 만족할 때만 `standingStillTime`을 증가시킨다.

* 전투 중  
* 게임 Scene이 일시정지 상태가 아님  
* 연구 기록 UI가 열려 있지 않음  
* 업그레이드 화면이 아님  
* 플레이어 이동 속도가 5 미만  
* 플레이어가 대시 중이 아님

# **43\. AI 분석 계산값**

`adaptiveAI.js`에서 다음 값을 안전하게 계산한다.

0으로 나누는 오류가 발생하지 않게 한다.

totalAttackUses \=  
  meleeUses \+ throwUses

meleeRatio \=  
  meleeUses / max(totalAttackUses, 1\)

throwRatio \=  
  throwUses / max(totalAttackUses, 1\)

combatMinutes \=  
  combatTime / 60

dashRate \=  
  dashUses / max(combatMinutes, 1\)

empUsageRate \=  
  empUses / max(empChargeCompletedCount, 1\)

empQuickUseRate \=  
  empUsesWithin8Seconds / max(empUses, 1\)

standingRatio \=  
  standingStillTime / max(combatTime, 1\)

averageEncounterDuration \=  
  totalCombatEncounterDuration /  
  max(combatEncounterCount, 1\)

분석 결과 유형은 다음 중 하나다.

* `melee`  
* `throw`  
* `dodge`  
* `emp`  
* `defensive`  
* `balanced`

남은 후보가 한 개면 한 개만 표시한다.

# **84\. 보스룸 구조**

보스룸은 화면보다 넓지만 지나치게 크지 않은 단일 전투 공간으로 구성한다.

1500×900

보스는 중앙에 배치한다.

보스룸에는 다음 물체를 배치한다.

금속 상자 3개

* 제거되거나 사용 불가능한 상태가 되면 8초 후 원래 위치에 재생성

* 보스 처치 후 재생성 타이머 제거

모든 공격은 플레이어가 경고를 보고 이동하거나 대시하여 피할 시간이 있어야 한다.

보스 공격 이벤트는 중앙 스케줄러 하나 또는 명확한 공격 루프에서 관리한다.

서로 다른 반복 타이머를 무제한으로 중첩하지 않는다.

# **88\. 보스 피해 처리**

플레이어의 근접 공격과 금속 상자 투척은 보스 중앙 코어에 피해를 줄 수 있다.

보스가 일반 상태일 때:

* 근접 피해 정상 적용  
* 투척 피해 정상 적용  
* EMP는 완전 정지가 아닌 짧은 과부하 적용

같은 공격 판정이 같은 보스에게 여러 프레임 반복 피해를 주지 않게 한다.

보스가 Phase 전환 연출 중일 때는 짧은 무적을 적용할 수 있다.

Phase 전환 무적 상태는 시각적으로 명확하게 표시한다.

Phase 전환이 끝나면 반드시 무적을 해제한다.

# **115\. ScoreManager**

점수 계산은 `scoreManager.js`에서 중앙 관리한다.

Scene과 Entity가 점수를 직접 더하지 않는다.

점수 지급 함수는 최소한 다음 값을 받는다.

eventType  
baseScore  
uniqueEventId  
isCombatScore

같은 `uniqueEventId`가 이미 처리되었다면 점수를 다시 지급하지 않는다.

점수 획득 계산:

baseScore  
× difficultyScoreMultiplier  
× dataCollectorMultiplier  
× combatComboMultiplier

`combatComboMultiplier`는 `isCombatScore === true`일 때만 적용한다.

배율 적용 후:

Math.round()

를 사용하여 정수화한다.

점수 획득 즉시 현재 점수에 반영하고 HUD를 갱신한다.

결과 화면에서 난이도 배율이나 데이터 수집기 배율을 다시 곱하지 않는다.

---

# **116\. 데이터 수집기 배율**

Part 2의 업그레이드 스택 수를 사용한다.

dataCollectorMultiplier \=  
1 \+ (0.15 × 데이터 수집기 스택 수)

최대 스택 2회이므로 최대 배율은 다음과 같다.

1.30

같은 점수 이벤트에 데이터 수집기 배율을 두 번 적용하지 않는다.

정확한 투척 공격 점수는 다음 조건을 만족할 때만 지급한다.

* 플레이어가 직접 던진 물체

* 방패에 완전히 차단된 공격이 아님

투척 적중 점수 파밍을 제한하기 위해 한 플레이에서 최대 20회까지만 지급한다.

보스룸의 재생성 금속 상자를 반복 사용해 무한 점수를 얻지 못하게 한다.

# **118\. 점수 감점**

다음 감점은 난이도 배율과 데이터 수집기 배율을 적용하지 않는다.

Stage 또는 Boss 재시작: \-500  
플레이어가 받은 실제 피해 1당: \-1

피해 감점은 실제 체력이 감소한 값만 적용한다.

무적 상태에서 차단된 피해는 감점하지 않는다.

감점 후 점수는 0 아래로 내려가지 않는다.

Stage 재시작 감점은 Part 2의 체크포인트 복원 후 한 번 적용한다.

# **119\. 콤보 시스템**

콤보는 적 또는 보스에게 실제 피해를 줄 때 증가한다.

하나의 피해 이벤트당 콤보 1 증가를 기본으로 한다.

하나의 공격으로 여러 적에게 피해를 주면 실제 피해를 받은 적 수만큼 증가할 수 있다.

콤보 유지 시간:

2초

2초 동안 적 또는 보스에게 피해를 주지 못하면 콤보를 0으로 초기화한다.

플레이어가 실제 피해를 받으면 현재 콤보를 절반으로 감소시킨다.

Math.floor(currentCombo / 2\)

콤보 배율:

0\~4 콤보: 1.0  
5\~9 콤보: 1.1  
10\~19 콤보: 1.2  
20\~29 콤보: 1.4  
30 이상: 1.6

콤보 배율은 다음 전투 점수에만 적용한다.

* 적 처치 점수  
* 보스 처치 점수  
* 정확한 투척 적중 점수

다음 점수에는 적용하지 않는다.

* 퍼즐 점수  
* 연구 기록 점수  
* 최종 보너스 점수

현재 플레이 최고 콤보는 `gameSession.js`에 저장한다.

# **120\. 최종 보너스 점수**

* 콤보 배율 미적용

final-low-damage

# **125\. 플레이 결과 확정**

`ResultScene` 최초 진입 시 다음 순서로 결과를 계산한다.

1. `gameSession.resultCommitted` 확인  
2. 최종 보너스 점수 계산  
3. 최종 점수 확정  
4. 최종 등급 계산  
5. 업적 조건 검사  
6. 최고 기록과 비교  
7. 영구 저장 데이터 갱신  
8. `gameSession.resultCommitted = true`

`resultCommitted`가 이미 true라면 영구 기록과 누적 통계를 다시 증가시키지 않는다.

ResultScene 재진입, 버튼 중복 입력, 브라우저 프레임 중복으로 결과가 여러 번 저장되지 않게 한다.

# **126\. 영구 기록 갱신**

결과 확정 시 다음 규칙을 사용한다.

## **최고 점수**

현재 최종 점수가 기존 최고 점수보다 높을 때만 갱신한다.

## **최고 등급**

현재 등급의 내부 순서 값이 기존 최고 등급보다 높을 때만 갱신한다.

## **최단 클리어 시간**

게임을 정상 클리어한 경우에만 비교한다.

기존 값이 `null`이면 현재 시간을 저장한다.

현재 시간이 더 짧을 때만 갱신한다.

## **최고 콤보**

현재 플레이 최고 콤보가 더 높을 때만 갱신한다.

## **난이도 클리어**

Normal 클리어:

normalCleared \= true

Hard 클리어:

hardCleared \= true

Easy 클리어는 별도 영구 Boolean을 추가하지 않는다.

## **누적 통계**

정상 결과 확정 시:

totalPlayCount \+= 1  
totalEnemyKills \+= 현재 플레이 적 처치 수

동일 결과를 두 번 반영하지 않는다.

# **129\. SettingsScene**

`SettingsScene.js`는 다음 경로에서 열 수 있다.

* `MenuScene`  
* `PauseScene`

returnScene: "MenuScene"

또는:

returnScene: "PauseScene"

설정 항목:

* 배경음 볼륨  
* 효과음 볼륨  
* 화면 흔들림 켜기·끄기  
* 전체 화면 켜기  
* 조작법 보기

MenuScene에서 설정을 열었을 때만 다음 항목을 표시한다.

* 저장 기록 초기화

PauseScene에서 설정을 열었을 때는 저장 기록 초기화 버튼을 표시하지 않는다.

설정 종료 시 전달받은 `returnScene`으로 돌아간다.

PauseScene에서 SettingsScene을 열었다면 아래 게임 Scene은 계속 일시정지 상태를 유지한다.

SettingsScene 종료가 게임 Scene을 직접 재개하지 않게 한다.

# **130\. 저장 기록 초기화**

확인 문구:

최고 기록, 업적, 누적 통계를 모두 초기화하시겠습니까?  
이 작업은 되돌릴 수 없습니다.

선택:

* 취소  
* 초기화

취소 시 아무 데이터도 변경하지 않는다.

초기화 확인 시:

* 최고 기록 초기화  
* 업적 초기화  
* 누적 통계 초기화  
* 마지막 난이도는 Normal로 초기화 가능  
* 볼륨과 화면 흔들림 설정은 유지

현재 플레이 중에는 초기화 기능을 제공하지 않는다.

# **131\. PauseScene**

PauseScene은 현재 게임 Scene 위에 병렬로 실행한다.

일시정지 가능한 Scene:

* ResearchLabScene  
* SecurityScene  
* ServerScene  
* BossScene

Esc 입력 시 다음 순서로 실행한다.

1. 현재 게임 Scene Key 확인  
2. PauseScene이 이미 실행 중인지 확인  
3. 현재 게임 Scene `scene.pause()`  
4. `PauseScene`을 다음 데이터와 함께 실행

pausedSceneKey

PauseScene 메뉴:

* 계속하기  
* 설정  
* 조작법  
* 메인 메뉴

계속하기:

1. PauseScene 종료  
2. `pausedSceneKey`에 해당하는 Scene 하나만 재개  
3. 입력 상태 초기화

설정:

1. PauseScene을 일시적으로 sleep 또는 pause  
2. SettingsScene 실행  
3. `returnScene: "PauseScene"` 전달  
4. 설정 종료 후 PauseScene 표시  
5. 아래 게임 Scene은 계속 정지 상태 유지

메인 메뉴:

* 확인 UI 표시  
* 확인하면 현재 플레이 종료  
* 게임 Scene과 PauseScene 정리  
* MenuScene 이동

PauseScene을 중복 실행하지 않는다.

# **132\. Esc 입력 우선순위**

Esc 입력은 다음 우선순위를 사용한다.

1. 연구 기록 UI가 열려 있으면 기록 UI 닫기  
2. 설정 팝업 또는 확인 팝업이 열려 있으면 해당 팝업 닫기  
3. PauseScene이 열려 있으면 게임 계속하기  
4. 일반 게임 Scene이면 PauseScene 열기

하나의 Esc 입력이 기록 UI를 닫는 동시에 PauseScene까지 열지 않게 한다.

# **134\. AudioManager**

오디오는 `audioManager.js`에서 중앙 관리한다.

외부 음원 파일을 사용하지 않는다.

Web Audio API를 사용하여 절차적 효과음과 간단한 반복 배경음을 생성한다.

브라우저 자동재생 정책을 고려한다.

첫 사용자 클릭 또는 키 입력이 발생했을 때 AudioContext를 생성하거나 resume한다.

AudioContext가 활성화되지 않은 상태에서 발생한 사운드 요청 때문에 게임이 멈추지 않게 한다.

필수 메서드 예:

unlockAudio()  
playSfx(key)  
playMusic(key)  
stopMusic()  
setBgmVolume(value)  
setSfxVolume(value)

같은 배경음이 중복 재생되지 않게 한다.

Scene 전환 시 기존 음악을 정리하고 필요한 음악만 재생한다.

# **146\. GitHub Pages 워크플로**

다음 파일을 생성한다.

.github/workflows/pages.yml

별도 빌드 없이 저장소의 정적 파일 전체를 배포한다.

워크플로는 공식 GitHub Pages Actions를 사용한다.

구성 요구사항:

* `actions/checkout@v4`  
* `actions/configure-pages@v5`  
* `actions/upload-pages-artifact@v3`  
* `actions/deploy-pages@v4`  
* Pages 쓰기 권한  
* id-token 쓰기 권한  
* `main` 브랜치 push 시 실행  
* 수동 실행 지원  
* 배포 artifact 경로는 저장소 루트

워크플로 예시 구조:

name: Deploy static site to Pages

on:  
  push:  
    branches:  
      \- main  
  workflow\_dispatch:

permissions:  
  contents: read  
  pages: write  
  id-token: write

concurrency:  
  group: pages  
  cancel-in-progress: true

jobs:  
  deploy:  
    environment:  
      name: github-pages  
      url: ${{ steps.deployment.outputs.page\_url }}

    runs-on: ubuntu-latest

    steps:  
      \- name: Checkout  
        uses: actions/checkout@v4

      \- name: Setup Pages  
        uses: actions/configure-pages@v5

      \- name: Upload artifact  
        uses: actions/upload-pages-artifact@v3  
        with:  
          path: .

      \- name: Deploy  
        id: deployment  
        uses: actions/deploy-pages@v4

저장소 이름에 관계없이 상대 경로가 정상 작동해야 한다.
